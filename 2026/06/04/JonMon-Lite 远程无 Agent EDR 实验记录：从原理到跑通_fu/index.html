前言

传统 EDR 的部署逻辑是：在每台端点上安装 Agent，Agent 在本机订阅 ETW（Event Tracing for Windows）Provider，收集进程创建、网络连接、文件操作等遥测数据后上报。这套模型行之有效，但存在几个固有局限：

  • 某些高价值服务器出于稳定性考量，运维团队不接受部署额外 Agent
  • Agent 本身是攻击面，存在被 Tamper 或卸载的风险
  • 大规模部署和维护 Agent 的工程成本较高

Huntress 研究员 Jonathan Johnson 在一次改写工具的过程中，意外发现 Windows 原生组件 Performance Logs and Alerts（PLA） 暴露了一套 DCOM 接口，允许通过网络远程创建 ETW 追踪会话，日志文件可直接回传至监控方机器——全程无需在目标机器上落地任何文件。

本文从原理到实践，系统梳理这一技术。

一、技术背景

1.1 ETW 简介

ETW（Event Tracing for Windows）是 Windows 内核级的事件追踪框架，由三个核心组件构成：

  • Provider：事件生产者，内核组件或用户态程序，负责发出事件（如 Microsoft-Windows-Kernel-Process）
  • Session：事件追踪会话，订阅若干 Provider，将事件写入 ETL 文件或实时消费
  • Consumer：读取和处理 ETL 文件或实时事件流的程序

几乎所有现代 EDR 都依赖 ETW 获取高质量的端点遥测数据。

1.2 COM 与 DCOM

COM（Component Object Model）是 Windows 的组件化接口规范，允许进程间通过接口调用暴露的功能。DCOM（Distributed COM）是其网络扩展——将 COM 调用搬到网络上，跨机器调用远程对象的方法。

WMI、PerfMon、以及本文主角 PLA，底层均使用 DCOM 实现远程访问能力。

1.3 发现过程

Johnson 在将 EtwInspector 从 C/C++ 改写为 .NET 版本时，需要实现枚举 ETW 追踪会话并关联其 Provider 的功能。他知道 logman.exe 具备这一能力，于是对其进行逆向分析。

逆向结果揭示了一个此前几乎未被安全社区关注的事实：logman 底层通过 COM 接口与 pla.dll 交互，而这些接口是 DCOM 接口，支持指定远程服务器。

顺着 IDataCollectorSetCollection::GetDataCollectorSets 这个接口继续挖掘，他发现 PLA 不仅支持枚举，还支持远程创建、修改和停止数据收集器——这构成了本文技术方案的完整基础。

二、PLA 核心概念

2.1 Data Collector 类型

PLA 将数据收集机制抽象为 Data Collector，共 5 种类型：

枚举值 编号 对应接口 用途
plaPerformanceCounter 0 IPerformanceCounterDataCollector 性能计数器
plaTrace 1 ITraceDataCollector ETW 追踪会话
plaConfiguration 2 IConfigurationDataCollector 注册表 / WMI / 文件提取
plaAlert 3 IAlertDataCollector 阈值告警
plaApiTrace 4 IApiTracingDataCollector API 调用追踪

其中 plaTrace 和 plaConfiguration 对安全研究最具价值，前者对应 ETW 追踪，后者可查询注册表、执行 WMI 查询、提取文件。

Data Collector Set 是一组 Data Collector 的容器，通过 IDataCollectorSet 接口管理，是 PLA 的主要操作对象。

2.2 命名空间

PLA 使用命名空间指定数据收集器的存储位置，主要包括：

命名空间 说明
Session\ 普通 ETW 追踪会话
AutoSession\ Autologger（随系统启动的追踪会话）
Service\ 用户定义的 Data Collector Set

关键发现：logman query -ets 默认仅查询 Session\ 命名空间。在 Service\ 命名空间下创建的追踪会话，不会出现在默认枚举结果中，具有一定的隐蔽性。

目标机器上以 SYSTEM 身份运行的追踪会话，会将 ETL 文件直接写入该路径。这意味着监控方无需主动拉取日志，数据自动回传。

认证方面，可通过 IDataCollectorSet::SetCredentials 指定运行账户，该账户需具备对 UNC 路径的写入权限。

3.4 权限要求

  • 在目标机器上创建 Data Collector Set，需要目标机的本地管理员权限
  • 创建后的 Collector Set 默认以 SYSTEM 身份运行
  • 若需写入远程 UNC 路径，运行账户需具备对应的网络共享权限

六、检测场景示例

场景一：检测 .NET 恶意程序集加载

监控 Microsoft-Windows-DotNETRuntime Provider，可捕获目标机器上所有 .NET Assembly 的加载行为。

攻击者通常会 Patch 本机内存中的该 Provider 以规避检测，但若完全不知道有远程收集在运行，则无从对抗。

场景二：检测 WMI 永久事件订阅

监控 Microsoft-Windows-WMI-Activity Provider，可捕获永久 WMI 事件订阅的创建，包括 Filter、Consumer、Binding 的完整细节——这是 APT 常用的无文件持久化手法。

场景三：检测 DCSync 攻击

监控 Microsoft-Windows-RPC Provider，可捕获 GetNCChanges RPC 调用，精确识别执行 DCSync 的进程及账户，即使攻击者使用 Mimikatz 内存注入方式。

七、进攻视角（供防御参考）

了解攻击者可能的利用方式，有助于构建针对性的检测规则。

侦察阶段：通过 IConfigurationDataCollector 远程查询注册表键值（如安全产品的安装路径、配置）、执行 WMI 查询、提取特定文件，全程无需在目标机器上运行任何进程或落地文件。

信息收集：创建远程 ETW 追踪会话，收集目标机器上的敏感遥测数据，如认证事件、加密操作、内核 API 调用等。

防御削弱：利用这些 DCOM 接口远程停止目标机器上安全产品创建的 ETW 追踪会话，或修改其会话配置，移除订阅的 Provider，使监控失效。

检测思路：监控 DCOM 对 PLA 接口的远程调用，关注以下事件：

  • IDataCollectorSet::Commit 远程调用（server 参数非空）
  • 非预期账户创建或修改 Data Collector Set
  • Service\ 命名空间下出现非预期收集器

八、与传统方案对比

维度 传统 Agent EDR WMI 远程执行 PLA DCOM 方案
目标机落地文件 是 通常需要 否
安全工具关注度 高 高 极低
ETW 数据丰富度 极高（本机） 无 极高（远程）
日志持续收集 是 难 是
部署复杂度 高 中 中
适用场景 通用端点 一次性命令执行 无法部署 Agent 的节点

九、总结

PLA DCOM 接口并非新技术，它一直以来都是 Windows 的原生能力，只是从未被安全社区系统审视过。其核心价值在于：

  • 无 Agent 的 ETW 采集：在不对目标机器产生任何落地影响的前提下，获取与本地 EDR 同等质量的遥测数据
  • 覆盖 EDR 盲区：对不适合部署 Agent 的服务器、网络设备提供监控能力
  • 攻击面意识：防御方需意识到，攻击者同样可以利用这套接口进行隐蔽侦察和防御干扰

这一发现的方法论同样值得关注：对成熟工具（logman）进行逆向分析，往往能在已知功能的背面发现未被关注的安全盲区。
